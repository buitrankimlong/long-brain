## LESSON 01 - Introduction to Generative AI and Large Language Models

### What is Generative AI?

Generative AI is artificial intelligence capable of generating text, images, and other content from natural language prompts. No programming language required - just natural language.

**Timeline of AI evolution:**
- 1960s: First AI prototypes - typewritten chatbots with keyword-triggered knowledge bases
- 1990s: Statistical approach - Machine Learning algorithms learn patterns from data
- 2000s-2010s: Neural networks / Deep Learning - RNNs power virtual assistants
- 2017+: Transformer architecture - overcomes RNN limits, enables modern LLMs

**Why Transformers changed everything:**
- Attention mechanism: assigns different weights to inputs regardless of order in sequence
- Can process much longer text sequences than RNNs
- Foundation for all modern LLMs (GPT, Claude, Gemini, etc.)

### How LLMs Work - Core Mechanics

**Step 1: Tokenization**
- LLMs don't read text - they read numbers (tokens)
- Tokenizer splits input text into tokens (variable-size chunks)
- Each token mapped to an integer index
- Different models tokenize differently

**Step 2: Token prediction**
- Given N input tokens, model predicts the next most likely token
- Output token added to input, process repeats (expanding window)
- This is why ChatGPT appears to "type" one word at a time

**Step 3: Probability + Temperature**
- Model predicts a probability distribution over all possible next tokens
- Does NOT always pick highest-probability token
- Temperature parameter controls randomness:
  - Temperature = 0: deterministic, always picks most likely token
  - Temperature = 1: very random, creative outputs
  - Default: 0.7

### Key Vocabulary
- **Prompt**: Text input sent to the LLM
- **Completion**: Text output generated by the LLM
- **Token**: Smallest unit the model processes (chunk of text -> number)
- **Temperature**: Parameter controlling output randomness (0-1 scale)

### What LLMs Can Do (Use Cases)
1. **Summarization** - articles, reviews, documents
2. **Creative writing** - essays, stories, marketing copy
3. **Conversation** - Q&A in natural language
4. **Text completion** - writing assistance
5. **Code generation and explanation**

### LLM Limitations (Critical to Know)
- Non-deterministic: same prompt may produce different outputs
- Can "hallucinate" (fabricate facts confidently)
- Not intelligent in the human sense - no critical reasoning, emotional intelligence
- Not trustworthy without verification - erroneous references combined with correct info

## LESSON 02 - Exploring and Comparing Different LLMs

### LLM Categories by Output Type

| Category | Purpose | Examples |
|----------|---------|---------|
| Audio/Speech | Speech recognition, multilingual | Whisper |
| Image Generation | Create/edit images | DALL-E 3, Stable Diffusion, Midjourney |
| Text/Code Generation | General language tasks | GPT-3.5, GPT-4, GPT-4o, LLaMA |
| Embeddings | Numerical text representation | OpenAI text-embedding-ada-002 |
| Multi-modal | Multiple input/output types | GPT-4 Turbo with Vision, GPT-4o |

### Foundation Models vs LLMs

**Foundation Models** (coined by Stanford, 2021):
- Trained with unsupervised/self-supervised learning on unlabeled multi-modal data
- Very large - billions of parameters
- Serve as base for downstream fine-tuned models
- Example: GPT-3.5 is a foundation model; ChatGPT is GPT-3.5 fine-tuned on conversation data

### Open Source vs Proprietary Models

**Open Source:**
- Inspectable, modifiable, customizable
- Examples: Alpaca (Stanford), Bloom (BigScience), LLaMA (Meta)
- Tradeoffs: may not be production-optimized, limited long-term maintenance

**Proprietary:**
- Owned by companies, optimized for production
- Examples: OpenAI GPT series, Google Bard, Claude (Anthropic)
- Tradeoffs: not inspectable, subscription cost, data privacy dependent on vendor

### Architecture Types

**Decoder-only** (most modern LLMs):
- Great at generating content
- Examples: GPT family (GPT-3, GPT-4)
- Analogy: The "content creator" - writes new content

**Encoder-only:**
- Great at understanding/classifying text
- Example: BERT
- Analogy: The "reviewer" - analyzes existing content

**Encoder-Decoder:**
- Can both understand and generate
- Examples: BART, T5
- Best for translation, summarization

### Service vs Model Distinction

- **Model**: The raw neural network (weights, parameters). Can run locally but requires hardware.
- **Service**: Cloud product wrapping a model + infrastructure. Pay-as-you-go, managed, scalable.
- **Azure OpenAI Service**: Enterprise-grade service with responsible AI framework on top of OpenAI models

### 3 Approaches to Improving LLM Results (Critical Decision Framework)

```
Cost/Complexity: Low -----> High
Approach:        PE  ->  RAG  ->  Fine-tuning  ->  Train from scratch
```

**1. Prompt Engineering with Context**
- Provide context, examples, instructions in the prompt
- Zero-shot (no examples), One-shot (1 example), Few-shot (2+ examples)
- Most cost-effective starting point

**2. Retrieval Augmented Generation (RAG)**
- Augment prompt with relevant external data (from vector DB or API)
- Overcomes LLM knowledge cutoff
- Good when: lack data/time/resources to fine-tune, want to reduce hallucinations
- Tools: Azure Vector Search, Pinecone, Weaviate, Redis

**3. Fine-tuning**
- Further train a model on your own data
- Creates a NEW model with updated weights
- Required when: need low-latency, have high-quality labeled data, very specific domain
- More expensive but more accurate for narrow tasks

**4. Training from Scratch**
- Most expensive, requires massive data and compute
- Only for domain-specific scenarios with vast domain data

## LESSON 03 - Using Generative AI Responsibly

### Microsoft's 6 Responsible AI Principles

1. **Fairness** - Free from bias and discrimination, treats everyone equally
2. **Inclusiveness** - Empowers everyone, accessible to all
3. **Reliability/Safety** - Performs reliably, safe for users
4. **Security & Privacy** - Secure systems, respects user data
5. **Transparency** - Understandable decisions and processes
6. **Accountability** - Humans remain accountable for AI systems

### Key Harms to Prevent

**Hallucinations:**
- LLM produces content that is factually wrong but presented confidently
- Example: "Who was the sole survivor of the Titanic?" - model invents a name
- Consequence: misleads users, damages product reputation

**Harmful Content:**
- Instructions for self-harm, hate speech, violent plans
- Instructions for illegal activities
- Sexually explicit content
- Must have tools/strategies to prevent this

**Lack of Fairness:**
- Reinforcing exclusionary worldviews
- Biased outputs that disadvantage certain groups
- Destructive to product and society

### 4-Step Responsible AI Process

```
1. MEASURE -> 2. MITIGATE -> 3. OPERATE -> 4. EVALUATE
```

**Step 1: Measure Potential Harms**
- Test with diverse prompts users will likely send
- For education app: cover subject questions, historical facts, student life prompts

**Step 2: Mitigate in 4 Layers**

| Layer | What It Means |
|-------|--------------|
| **Model** | Choose right model for use case; fine-tune to reduce harmful outputs |
| **Safety System** | Platform-level content filtering (Azure AI Content Safety); detect jailbreaks |
| **Metaprompt** | System instructions limiting model behavior; RAG to restrict info sources |
| **User Experience** | UI/UX design limiting harmful inputs; transparency about AI capabilities |

**Step 3: Operate Responsibly**
- Partner with Legal and Security teams
- Build delivery, incident handling, and rollback plans before launch

**Step 4: Evaluate Model**
- Measure accuracy, similarity, groundedness, relevance
- Even without control over training data, always evaluate outputs

### Tools
- **Azure AI Content Safety**: API to detect harmful content and images
  - https://learn.microsoft.com/azure/ai-services/content-safety/overview

## LESSON 04 - Prompt Engineering Fundamentals

### What is Prompt Engineering?

> The process of designing and optimizing text inputs (prompts) to deliver consistent, quality responses for a given application objective.

Two-step process:
1. **Design** - construct initial prompt for the model and objective
2. **Refine** - iterate to improve response quality

It is currently more **art than science** - requires intuition, trial-and-error, domain expertise.

### Why Prompt Engineering Matters

1. **Model responses are stochastic** - same prompt produces different outputs
2. **Models fabricate responses** - hallucinate facts, invent references
3. **Model capabilities vary** - newer models have different quirks and tradeoffs

### Core Concepts

**Tokenization:** LLMs see prompts as sequences of tokens (numbers), not raw text. How you write affects how it tokenizes, which affects output quality.

**Base LLMs (Foundation Models):** Predict next token statistically - don't "understand" meaning, just pattern-match. Can complete sequences.

**Instruction-Tuned LLMs:** Fine-tuned with examples using RLHF (Reinforcement Learning with Human Feedback) to follow explicit instructions, not just complete text.

### Prompt Construction - Components

**1. Basic Prompt:**
```
Oh say can you see
-> Completes "The Star-Spangled Banner" lyrics
```

**2. Complex Prompt with Chat API:**
```python
response = openai.chat.completions.create(
    model="gpt-3.5-turbo",
    messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)
```
Three roles: `system` (sets context/behavior), `user` (human input), `assistant` (AI response)

**3. Primary Content Pattern:**
```
[Instruction] + [Content to act on]

Example:
Jupiter is the fifth planet... [long text]
**Summarize this in 2 short sentences**
```

### Prompting Techniques

**Zero-shot:** Single prompt, no examples
```
"The Sun is Shining". Translate to Spanish
-> "El Sol está brillando"
```

**One-shot:** Single example provided
```
"The Sun is Shining" => "El Sol está brillando"
"It's a Cold and Windy Day" => [model infers pattern]
```

**Few-shot:** Multiple examples
```
The player ran the bases => Baseball
The player hit an ace => Tennis
The player hit a six => Cricket
The player made a slam-dunk => Basketball [model infers]
```

**Prompt Cues:** Start the completion you want, model continues in that direction
```
Summarize This
Top 3 Facts We Learned:    <- cue forces list format
```

**Prompt Templates (LangChain style):**
```python
# Variables replaced at runtime
from langchain.prompts import PromptTemplate
template = "Summarize {text} for a {audience}"
```

### Best Practices Table

| Practice | Why |
|---------|-----|
| Evaluate latest models | New generations have improved features |
| Separate instructions from context | Helps model assign weights accurately |
| Be specific and clear | Improves quality and consistency |
| Use examples (show and tell) | Start zero-shot, refine with few-shot |
| Use cues to start completions | Nudge toward desired format |
| Repeat important instructions | Model may need reinforcement |
| Order matters | Recency bias affects output |
| Give the model an "out" | Reduces fabrications when model uncertain |

**Prompt Engineering Mindset:**
1. **Domain Understanding** - customize for your specific domain
2. **Model Understanding** - know strengths/limits of the model you use
3. **Iterate & Validate** - build a prompt library/knowledge base

- Started with OpenAI Codex model
- Rapidly needed fine-tuning + better prompt engineering to improve code quality
- July 2023: debuted improved AI model beyond Codex
- Key insight: prompt engineering + model selection = production-ready AI feature

## LESSON 05 - Advanced Prompting Techniques

### All Prompting Techniques Summary

| Technique | Description | Best For |
|-----------|-------------|---------|
| Zero-shot | Single prompt, no examples | Simple/general tasks |
| Few-shot | 1+ examples provided | Tasks needing style/format guidance |
| Chain-of-thought | Step-by-step reasoning shown | Math, logic, complex reasoning |
| Generated knowledge | Inject company/domain data | Business-specific responses |
| Least-to-most | Break problem into subproblems | Complex multi-step tasks |
| Self-refine | Ask LLM to critique + improve its output | Code, essays, plans |
| Maieutic prompting | Ask LLM to explain each part of its answer | Verifying answer consistency |

### Chain-of-Thought (CoT) - Deep Dive

**Problem without CoT:**
```
Prompt: "Alice has 5 apples, throws 3, gives 2 to Bob, Bob gives 1 back. How many?"
Answer: 5  (WRONG - correct is 1)
```

**With Chain-of-Thought:**
```
Prompt: "Lisa has 7 apples, throws 1, gives 4 to Bart, Bart gives 1 back:
  7 - 1 = 6
  6 - 4 = 2
  2 + 1 = 3

Alice has 5 apples, throws 3, gives 2 to Bob, Bob gives 1 back. How many?"
Answer: 1  (CORRECT)
```

Key: Show a worked example with explicit calculation steps BEFORE the actual problem.

### Generated Knowledge - Template Pattern

Use company data to fill prompt templates:

```text
{{company}}: {{company_name}}
{{products}}:
{{products_list}}
Please suggest an insurance given the following budget and requirements:
Budget: {{budget}}
Requirements: {{requirements}}
```

Filled template becomes:
```text
Insurance company: ACME Insurance
Insurance products (cost per month):
- type: Car, cheap, cost: 500 USD
- type: Car, expensive, cost: 1100 USD
- type: Home, cheap, cost: 600 USD

Please suggest an insurance given the following budget and requirements:
Budget: $1000 restrict choice to types: Car, Home
```

Lesson: Adding structure keywords like `type:`, `cost:`, and `restrict` dramatically improves accuracy.

### Self-Refine Technique

**Step 1:** Initial prompt
```
> Prompt: "Create a Python Web API with routes products and customers"
```

**Step 2:** AI generates code (may have issues)
```python
import flask
app = Flask(__name__)  # BUG: should be 'from flask import Flask'

@app.route('/products')
def products():
    return 'Products'
```

**Step 3:** Ask for critique
```
> Prompt: "suggest 3 improvements of the above code"
```

**Step 4:** AI improves code
```python
from flask import Flask, request

app = Flask(__name__)

@app.route('/products')
def products():
    return f'This is the {request.path} page'

if __name__ == '__main__':
    app.run()
```

Self-refine is extremely useful for code quality improvement in a chat loop.

### Maieutic Prompting (Verify Consistency)

1. Ask LLM to answer a question
2. For each part of the answer, ask LLM to explain in depth
3. Check for inconsistencies - discard inconsistent parts
4. Repeat until satisfied

Useful for: complex plans, research summaries, any output where accuracy matters.

### Temperature - Controlling Output Variance

```python
completion = client.chat.completions.create(
    model=deployment,
    messages=messages,
    temperature=0.1  # More deterministic
)
```

| Temperature | Behavior | Use Case |
|-------------|----------|---------|
| 0.0 | Highly deterministic, consistent outputs | Code generation, factual Q&A |
| 0.1-0.3 | Mostly consistent with minor variation | Technical docs, structured data |
| 0.7 | Default, balanced creativity/consistency | General chat, content |
| 0.9-1.0 | Highly varied, creative outputs | Brainstorming, creative writing |

Other parameters (advanced, out of scope for this course): `top-k`, `top-p`, `repetition_penalty`, `length_penalty`, `diversity_penalty`

### Good Practices Summary

- **Specify context** - domain, topic, background
- **Limit output** - specify exact count, length, format
- **Specify WHAT and HOW** - e.g. "Create Python Web API, divide into 3 files"
- **Use templates** - for repeatable prompts with company data
- **Spell correctly** - LLMs produce better output for well-formed inputs

## LESSON 08 - Building Search Applications with Embeddings

### Semantic Search vs Keyword Search

**Keyword Search:** Literal match - searching "dream car" finds documents about dreaming about cars.

**Semantic Search:** Understands meaning/intent - searching "dream car" finds "ideal car to buy" because it understands the semantic intent.

### What are Text Embeddings?

Text embeddings are **numerical vector representations of text** that capture semantic meaning.

```python
# Input text:
"Today we are going to learn about Azure Machine Learning."

# Output: vector of 1536 numbers (OpenAI text-embedding-ada-002)
[-0.006655829958617687, 0.0026128944009542465, 0.008792596869170666, ...]
```

Properties:
- Similar meaning = vectors are "close" in the vector space
- Dissimilar meaning = vectors are "far" from each other
- Distance measured by cosine similarity

### Cosine Similarity

```
cos(θ) = (A · B) / (||A|| * ||B||)
```

- Value range: -1 to 1 (for normalized vectors: 0 to 1)
- 1 = identical semantic meaning
- 0 = completely unrelated
- Advantage over Euclidean distance: works well even when document sizes differ

### Building an Embedding Index (Production Pipeline)

The course's YouTube search app used this pipeline:

1. **Download** transcripts for each YouTube video
2. **Extract** speaker names using OpenAI Functions (from first 3 min)
3. **Chunk** transcripts into **3-minute segments** (~20 word overlap between segments)
4. **Summarize** each chunk to 60 words with OpenAI Chat API
5. **Embed** each chunk with OpenAI Embedding API (returns 1536-dim vector)
6. **Store** `embedding_index_3m.json` (or production: vector database)

### Vector Databases (Production)

| Database | Notes |
|----------|-------|
| Azure Cognitive Search | Microsoft managed, enterprise-grade |
| Redis | High performance, open source |
| Pinecone | Purpose-built vector DB, fully managed |
| Weaviate | Open source, supports multiple modalities |
| pgvector (PostgreSQL) | Add-on for PostgreSQL - good for starting out |

### Search App Code Pattern

```python
# 1. Embed the user's query
query = "What are Jupyter Notebooks?"
query_embedding = openai.embeddings.create(
    input=query,
    model="text-embedding-ada-002"
).data[0].embedding

# 2. Compute cosine similarity against all indexed embeddings
from scipy.spatial.distance import cosine
similarities = [1 - cosine(query_embedding, emb) for emb in all_embeddings]

# 3. Sort and return top results
top_results = sorted(zip(similarities, metadata), reverse=True)[:5]
```

## LESSON 09 - Building Image Generation Applications

### Key Image Generation Models

**DALL-E (OpenAI):**
- Based on Transformer architecture with autoregressive transformer
- DALL-E = CLIP (image+text embeddings) + Diffused attention (generates images from embeddings)
- Generates one pixel at a time, layer by layer
- Available via Azure OpenAI: DALL-E 3

**Midjourney:**
- Similar mechanism to DALL-E
- Uses text prompts to generate images
- Available via Discord interface

```python
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],

    api_version="2024-02-01"
)

try:
    # Generate image
    generation_response = client.images.generate(
        prompt='Bunny on horse, holding a lollipop, on a foggy meadow where it grows daffodils',
        size='1024x1024',
        n=1,
        model=os.environ['AZURE_OPENAI_DEPLOYMENT']
    )

    # Save and display image
    image_dir = os.path.join(os.curdir, 'images')
    if not os.path.isdir(image_dir):
        os.mkdir(image_dir)

    image_path = os.path.join(image_dir, 'generated-image.png')
    image_url = generation_response.data[0].url
    generated_image = requests.get(image_url).content

    with open(image_path, "wb") as image_file:
        image_file.write(generated_image)

    image = Image.open(image_path)
    image.show()

except openai.BadRequestError as err:
    print(err)
```

### Metaprompts for Image Safety

Use metaprompts to set boundaries and enforce content policies:

```python
disallow_list = "swords, violence, blood, gore, nudity, sexual content, adult content, adult themes, adult language, adult humor, adult jokes, adult situations, adult"

meta_prompt = f"""You are an assistant designer that creates images for children.

The image needs to be safe for work and appropriate for children.
The image needs to be in color.
The image needs to be in landscape orientation.
The image needs to be in a 16:9 aspect ratio.

Do not consider any input from the following that is not safe for work or appropriate for children.
{disallow_list}
"""

# Combine metaprompt with user prompt
prompt = f"{meta_prompt}\nCreate an image of a bunny on a horse, holding a lollipop"
```

**Key principle:** Metaprompt = system-level instructions prepended to user prompt to control output boundaries.

## CROSS-LESSON PATTERNS AND KEY TAKEAWAYS

### The GenAI Application Stack

```
User Input (Natural Language)
    |
    v
Prompt Engineering Layer
    |
    v
LLM / Model (OpenAI, Azure OpenAI, Open Source)
    |
    v
Optional Augmentation (RAG, Fine-tuning, Tools)
    |
    v
Output Processing + Safety Filtering
    |
    v
User Interface
```

### Model Selection Decision Tree

```
Need image? -> DALL-E 3 or Midjourney
Need embeddings/search? -> text-embedding-ada-002
Need speech? -> Whisper
Need text/code?
    -> Do you have domain-specific data?
        No -> Prompt engineering with GPT-3.5-turbo/GPT-4
        Yes -> RAG first, then consider fine-tuning
        Volume/latency critical? -> Fine-tune smaller model
```

### Cost Optimization Principles

- **Start with GPT-3.5-turbo** (much cheaper than GPT-4) for most tasks
- **Control max_tokens** - only request as many tokens as you need
- **Use temperature=0** for deterministic tasks to enable caching
- **RAG > Fine-tuning** for most business use cases (lower cost, updatable)
- **Semantic caching**: cache embedding lookups to avoid redundant API calls

### Security Best Practices Across All Lessons

1. NEVER hardcode API keys - always use environment variables
2. Use `.env` files with `python-dotenv`, add `.env` to `.gitignore`
3. Implement content filtering (Azure AI Content Safety)
4. Use metaprompts as guardrails for user-facing applications
5. Validate and sanitize all user inputs before they reach the LLM

### Python Code Template for Any OpenAI App

```python
import os
from openai import AzureOpenAI
from dotenv import load_dotenv

load_dotenv()

client = AzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-02-01"
)

def call_llm(messages: list, max_tokens: int = 500, temperature: float = 0.7) -> str:
    response = client.chat.completions.create(
        model=os.environ["AZURE_OPENAI_DEPLOYMENT"],
        messages=messages,
        max_tokens=max_tokens,
        temperature=temperature
    )
    return response.choices[0].message.content

def call_embedding(text: str) -> list:
    response = client.embeddings.create(
        input=text,
        model="text-embedding-ada-002"
    )
    return response.data[0].embedding
```
